Add Hotspot — Design Documentation

🎓 Hotel Training Feature · Last updated April 2026
ℹ
This document covers the Add Hotspot step of the simulation creation wizard — where admins configure interactive areas on training screenshots that agents will interact with during a simulation run.

What is this tool?

The Add Hotspot tool lets training admins place and configure interactive hotspots on top of simulation screenshots. Each hotspot defines what the agent is expected to do on that part of the screen, how it is scored, and what happens next in the simulation flow.

The tool is a 3-column layout: a screen list on the left, the canvas in the centre where hotspots are drawn, and the Hotspot Properties panel on the right.

Information Architecture

The tool is divided into three columns that work together:

Left — Screen Navigation
Screen navigation items
Admin-defined name, truncated with tooltip. Status dots show correct/wrong/terminal paths.
Chain icon
Shown when screen is connected via navigation.
Legend
Colour key for the status dots. Fixed at bottom.
Centre — Canvas
Screenshot preview
Full training screen image. Scrollable for tall screenshots.
Hotspot overlays
Draggable, resizable rectangles. Colour-coded by type outcome.
Type selector + Add Hotspot
Anchored below canvas. Hidden during editing.
Right — Properties Panel
Browse state
Lists all hotspots on current screen + Current Screen Flow.
Edit state
Full form for name, scoring, action, and flow preview.
Save / Discard / Delete
Fixed action bar at the bottom of the panel.
⚠
Max hotspots per screen: 10. The Add Hotspot button is disabled and shows "Max Reached" when this limit is hit.

Hotspot Types

There are four hotspot types. Each behaves differently on the canvas and has different properties in the edit panel.

ads_click
Click Area
Marks a clickable target

The agent is expected to click within this area. Can reward a correct click, penalise a wrong click, or trigger terminal failure.

correct +pts wrong −pts ■ terminal
text_fields

Alphanumeric input

The agent must type a value. Admin sets an answer key. Scored on whether the final input matches.

conditional ■ terminal
check_box
Checkbox
Tick or untick

Admin sets the Expected End State (must be checked or unchecked). Scored on final state match — not on whether the agent clicked.

conditional ■ terminal
format_list_bulleted
Dropdown
Select from list

Admin defines dropdown items and marks the correct answer. Scored on whether the agent selects the right option.

conditional ■ terminal

Click Area — Properties

Click Area is the only non-conditional type. The outcome is determined by the admin toggling a correct or wrong click, and selecting what action follows.

Points

Admin toggles + or − using the sign prefix next to the points input. This determines whether the hotspot is a correct click or a wrong click.

ℹ
Points input accepts values from 0 → ∞, no error states. The + / − sign is a prefix icon — admin always inputs a regular number. If the admin enters an out-of-range value (e.g. a decimal), it auto-reverts to 1 (default). Uses an Input Number component with formatter + parser.

Action (Correct click-area)

Navigate to ScreenAdmin selects any screen after the current one. Defaults to the next screen in sequence.

Action (Wrong click-area)

Navigate to ScreenAdmin selects any screen after the current one. Backward navigation is not allowed.
Continue (silent)Points deducted but the simulation continues without interruption. Agent does not see feedback.
End SimulationTerminal Failure. Immediately ends the simulation, marks it as FAILED, sets score to 0%.
⚠
The Continue and End Simulation options are only shown when the click is set to wrong (− toggle). Correct clicks always navigate.

Text Field — Properties

Text Field is a conditional type. It always has two scoring paths: correct answer and wrong answer. The correct path always continues — there is no navigation choice.

Answer Key

Optional. Admin enters the expected text value the agent should type. If left empty, any input is treated as correct.

On Correct Answer

Points1 or more pts awarded (0 → ∞, default 1). Agent always continues to next hotspot.
ℹ
Points accept values from 0 → ∞. The + / − sign is a prefix icon set by the path (correct vs wrong) — admin always inputs a regular number. Out-of-range input (e.g. decimal) auto-reverts to 1. Uses Input Number with formatter + parser.

On Wrong Answer

Points1 or more pts deducted (0 → ∞, default 1). Disabled if terminal.
Continue (silent)Points deducted, simulation continues without feedback.
End SimulationTerminal Failure — ends immediately.

Checkbox — Properties

Checkbox is a conditional type with an important distinction: the admin sets the Expected End State, not the expected action. The agent is scored on whether the checkbox is in the right state at the end — not whether they clicked it.

Expected End State

☑ Must checkedThe checkbox must be checked when the agent finishes. If it's already checked, the agent should leave it alone.
☐ Must uncheckedThe checkbox must be unchecked when the agent finishes. If it's already unchecked, leave it alone.

Scoring matrix

Screenshot shows Expected End State Agent should Doing nothing is Example
☐ ☑ Must checked Click to check it ✗ Incorrect "Send me promotional emails" — opt-in unchecked by default, agent must check it
☑ ☐ Must unchecked Click to uncheck it ✗ Incorrect "Add travel insurance (+$49)" — pre-checked upsell the agent should remove
☐ ☐ Must unchecked Leave it alone ✓ Correct "I agree to share my data with partners" — already unchecked, agent recognises it's correct
☑ ☑ Must checked Leave it alone ✓ Correct "I agree to Terms & Conditions" — required checkbox already ticked, agent should not uncheck it
💡
Use rows 3 & 4 to test that an agent knows not to touch a checkbox that's already in the correct state — this tests restraint, not just action.

Dropdown — Properties

Dropdown is a conditional type. Admin builds the list of options and marks one as the correct answer.

Dropdown Items

Admin adds up to 15 items. Each item is a text string. Items can be deleted individually. The list populates the Answer Key selector below it.

Answer Key

Admin selects which item is the correct answer from the Answer Key dropdown. The selector is always visible — when no items have been added yet, it shows a "Select answer key" placeholder and is not interactive. Once at least one item is added, the correct answer can be selected.

On Correct / Wrong

Same branching as Text Field: correct always continues with +pts; wrong can either continue silently with −pts, or trigger Terminal Failure.

Navigation Rules

The simulation is linear and cannot loop backward. This affects all navigation dropdowns across hotspot types.

1
The Navigate to Screen dropdown only lists screens after the current one. Screens that come before are hidden entirely.
2
Both correct and wrong click-area hotspots let the admin freely pick any forward screen as the destination.
4
Conditional types (Text Field, Checkbox, Dropdown) always continue on the correct path. Navigation choices are not available for the correct path.
5
Result screens (Pass / Fail) appear as "Result Screen" in all navigation dropdowns and flow displays.

Scoring & Branching

Every hotspot produces one of three possible outcomes:

✓ Correct path
  • Awards points — the + prefix is added automatically; admin inputs a regular number (0 → ∞, default 1)
  • Simulation continues to next hotspot
  • Click Area: navigates to the next screen
✕ Wrong path
  • Deducts points — the − prefix is added automatically; admin inputs a regular number (0 → ∞, default 1)
  • Simulation continues silently
  • Agent receives no feedback
■ Terminal
  • Ends simulation immediately
  • Marks session as FAILED
  • Sets result score to 0%

Current Screen Flow

The Current Screen Flow section appears at the bottom of the Hotspot Properties panel. It shows a visual summary of all hotspots on the current screen that have a navigation or terminal impact.

What appears in the flow

ShownHotspots that navigate to another screen, or that can trigger Terminal Failure.
HiddenPure "continue" hotspots — they have no flow impact and would add noise.

How conditional types appear

Text Field, Checkbox, and Dropdown show two rows — one for the correct path and one for the wrong path — but only when at least one path has a navigation or terminal impact. If both paths just continue (no navigation, not terminal), neither row is shown, since neither has any flow impact.

The action bar at the bottom of the panel changes depending on whether the hotspot has been saved before.

StateLeft buttonRight buttonNotes
New, unfilled Discard Save Hotspot (disabled) Hotspot name is empty — Save remains disabled
New, filled Discard Save Hotspot Discard removes the unsaved hotspot from canvas
Editing saved Delete Hotspot Save Hotspot Delete permanently removes the hotspot; Save updates it
